# Implementation Notes: 087

### Format example

```yaml
items:
  076:
    title: brrr review manifest integration
    complexity: low
    scope: [{path: skills/brrr/SKILL.md, op: modify}]
    depends: []
    blocks: [080]
    criteria:
      - brrr Phase 6b generates review-manifest.md before spawning reviewers
      - all 3 reviewer prompts reference manifest not plan/work-items/*.md
      - journal-keeper prompt references manifest as index
    notes: plan/notes/076.md
```

Keys are human-readable. Scope ops: `modify`, `create`, `delete`. Criteria as plain strings (no checkbox syntax). Notes in separate files, loaded only by the executing worker.

### Validation tool (MVP)

`scripts/validate-specs.sh` subcommands: `dag` (cycle detection), `overlap` (scope conflicts between concurrent items), `coverage` (union of scopes vs expected files), `groups` (topological sort for execution strategy), `lint` (flag vague criteria terms). Viewer/renderer deferred.

Implementation: Python with `pyyaml`, or shell with `yq`.

### Skill changes

- **Plan**: write to `work-items.yaml` + `plan/notes/{id}.md` instead of per-file markdown
- **Execute**: single Read of `work-items.yaml` + per-worker Read of notes file (replaces N glob+read calls)
- **Review**: `work-items.yaml` replaces manifest generation — the YAML already IS a manifest
- **Refine**: Phase 3 read path updated to load from `work-items.yaml`. Phase 7h appends new entries and reads highest ID from YAML (not from globbing markdown files)

### Migration (extend 085)

Parse existing `plan/work-items/*.md` files, extract structured fields to YAML, move implementation notes to `plan/notes/`. Move originals to `plan/work-items-legacy/`.

### Backward compatibility

Skills check: if `work-items.yaml` exists, use it; otherwise fall back to `plan/work-items/*.md`.
